Deploying on AWS

Creating a simple static website with an S3 bucket was relatively simple. I uploaded the relevant files and enabled it for static website hosting. After purchasing my domain name and configuring route 53 and CloudFront to point my domain to my website, it still did not propagate after several hours. I discovered the name servers of my hosted zone and my registered domain did not match. Once I fixed this, my domain resolved properly and the previously requested HTTPS certificate instantly worked as well. Once I checked my root domain properly redirects and HTTPS was required, along with continually tweaking my website code, it was time to implement some basic web functions.

Automating with Terraform

I chose to use Terraform rather than SAM since it's a bit more flexible with other vendors. I started the process by deploying each service at a time, starting with the buckets. After declaring what types my uploaded files should be and setting an access policy, it went smoothly. Deploying CloudFront also had little issues, but properly getting my HTTPS certificate and DNS records was a bit trickier. I found it easier to do repeat steps manually to better visualize the steps needed for my IaC. After creating the records needed for the DNS verification, I realized I had the same issue as before" my name servers did not match. I changed them manually to verify this was the issue and then discovered they recently added a function to update my registered domain name servers. After automating this with Terraform's for loop equivalent, I was pleased to see it update in real time and for my certificate automatically validated.

Passing API URLs from Terraform

An Issue I discovered from Terraform automation is the API gateway URLs changed every deployment, which would break the Javascript lambda calls unless they were manually changed each time. I decided to resolve this in a Cloud CI/CD way:

  • Have Terraform create a JSON file with the API gateway invoke URLs
  • Store the JSON file in an S3 bucket
  • Have the Javascript retrieve the JSON file and use the URLs to invoke the Lambda Function
I had the fetches happened synchronously, since they depended on the JSON fetch, which increased the load time. I also had the Lambda display not load until they were completed, instead display incrementing loading text.

GitHub Actions

GitHub Actions executes a workflow when a specific event occurs. My workflow updates the files of the S3 bucket when I push changes to my repository. This way my website updates in real time whenever I make changes, instead of having to rerun my Terraform code.

A large reason for testing is to ensure that an unexpected input is handled properly and, as the joke eloquently demonstrates, it's not as simple as it might seem. However, my functions call upon values in a database, not directly from the user or code. So, while I can test for basic functionality and proper output, the major purpose of testing for abnormal inputs is unnecessary. But I did approach it thoroughly to ensure I can scale up my lambda functions and their tests for future additions. After some research, I discovered a package that enables me to run my Lambda functions locally, lambda_local. I also used moto to create mock AWS resources for my testing. While it's not necessary since my functions do not heavily manipulate the table data, I wanted to continue the trend of learning new technologies when possible. Since I was creating my mock tables during testing, I was able know exactly what the table data was, whether a static value or the current Unix time, and test whether the output properly incremented the count or correctly calculated the time difference.

Error Page

To replace the generic 404 and 403 error pages, I created a custom error page complete with a Javascript function to display the current status of the website. The Javascript calls on a Python Lambda Function to ensure compatibility, as client-side Javascript checks are sometimes blocked. I contemplated having this error page be hosted in a separate S3 bucket, in case I had torn down the website using Terraform. However, Terraform would also destroy the CloudFront distribution, which would cause the error page to be inaccessible anyways.

Automating Cache Invalidation

CloudFront caches the website files at Edge Locations to reduce latency and load. While this is great, since it prevents users from accessing the S3 bucket website directly, it does need to be told when I update the S3 files. While I could do this manually via the AWS GUI, I utilized what I learned to automate this process. First I simply found the AWS CLI commands to do and test this. While I could have just make a simple script and invoked it using GitHub actions, I decided to do this via a Lambda Function. I implemented a Python function to list my CloudFront distributions and invoke a call to invalidate all files in the wild card path /*. After testing, I then:

  • Created an API gateway linking to the clearCache Lambda Function.
  • Added the API URL to my JSON File/URL bucket, to ensure it works between Terraform deployments.
  • Used GitHub actions to parse through the JSON file and do a simple HTTP GET request to the API URL, clearing my cache on every git push.

I also made sure to check I wouldn't accidentally go out of the AWS free tier. But as long as I don't push to my Repository more than 10,000 times a month, it will stay free.
